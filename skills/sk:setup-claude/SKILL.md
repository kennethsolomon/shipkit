# /sk:setup-claude

Bootstrap or repair Claude Code infrastructure on any project in a safe, idempotent way.

## Workflow Contract (What This Enables)

After bootstrapping a project, the recommended workflow becomes:

1. `/sk:setup-claude` â€” ensure scaffolding exists
2. `/sk:brainstorm` â€” clarify design (no coding yet)
3. `/sk:write-plan` â€” write a decision-complete plan into `tasks/todo.md`
4. `/sk:execute-plan` â€” implement in small batches while logging progress
5. `/sk:security-check` â€” audit changed files for vulnerabilities and production quality
6. `/sk:review` â€” self-review (report-only). Loop `/sk:debug` + `/sk:smart-commit` until clean
7. `/sk:finish-feature` â€” finalize branch (changelog, arch log auto-committed, security gate, create PR)

## Critical Behavior Rules

### Read-before-decide
Before major decisions or after many tool calls, re-read `tasks/todo.md` to refresh the goal and constraints.

### Log errors with attempts (3-strike protocol)
When anything fails, log it immediately (error + attempt # + resolution). If the same approach fails twice, change approach. After 3 failures, stop and ask the user with what you tried.

### Always write findings/progress to disk
If you discover anything important (stack details, key paths, constraints, failures), write it into `tasks/findings.md` or `tasks/progress.md` before moving on.

### Lessons Capture (lessons.md)

`tasks/lessons.md` is a permanent log of corrections. Two triggers:

**Explicit** â€” user says any of these phrases:
- `lesson:` / `remember:` / `don't do this again:` / `add to lessons`
â†’ Immediately append to `tasks/lessons.md` without asking.

**Implicit** â€” detect correction language in user messages:
- Keywords: "no", "don't", "instead", "wrong", "not like that", "stop doing"
â†’ Ask: *"Should I add this to lessons.md?"* â†’ append on confirmation only.

Entry format:
```markdown
### [YYYY-MM-DD] [Brief title]
**Mistake:** What went wrong (symptom)
**Root cause:** Why it happened
**Prevention:** What to do differently next time
```

Never remove lessons unless the user explicitly asks to remove a specific one.
Never overwrite `tasks/lessons.md` â€” always append.

## Phase 0: Reconnaissance (first setup only)

On **first-time setup** (no existing `CLAUDE.md` or `tasks/findings.md`), run a reconnaissance pass before stack detection:

1. **Directory scan** â€” list top 2 levels of the project tree (excluding node_modules, vendor, .git, dist, build)
2. **Entry point detection** â€” find main.*, index.*, app.*, server.*, manage.py, artisan, Makefile
3. **Architecture classification**:
   - Monorepo: multiple package.json/go.mod files, `packages/` or `apps/` directory
   - Monolith: single manifest, `src/` or `app/` directory
   - Microservices: `services/` directory with separate manifests
4. **Data flow trace** â€” from entry point, identify: request handling â†’ validation â†’ business logic â†’ data persistence
5. **Output** â€” append architecture summary to `tasks/findings.md`:
   ```
   ## Reconnaissance (auto-generated by /sk:setup-claude)
   - Architecture: [monolith/monorepo/microservices]
   - Entry points: [list]
   - Key directories: [list with purposes]
   - Data flow: [request â†’ ... â†’ response]
   ```

Skip this phase on re-runs (when `tasks/findings.md` already contains "Reconnaissance").
