# TODO — 2026-03-23 — ShipKit Workflow Improvements (Hooks, Rules, Statusline, New Skills)

## Goal

Add 6 infrastructure improvements to ShipKit inspired by claude-code-game-studios:
1. **Lifecycle hooks** — auto-load context on session start, preserve state on compact, validate commits/pushes, log agents
2. **Path-scoped rules** — contextual rules per directory pattern, generated by stack detection
3. **Statusline** — persistent CLI status showing workflow step, branch, task, context %
4. **Scope check skill** (`/sk:scope-check`) — detect scope creep mid-implementation
5. **Retrospective skill** (`/sk:retro`) — post-ship analysis of velocity, blockers, patterns
6. **Reverse document skill** (`/sk:reverse-doc`) — generate docs from existing code

## Constraints (from lessons.md)

- All commands use `/sk:` prefix
- When adding new commands: update CLAUDE.md, README.md, DOCUMENTATION.md, install.sh, CHANGELOG.md
- Never overwrite `tasks/lessons.md` — append only
- Each new skill needs a lessons.md entry tracking its dependent files
- Templates use `<!-- Generated by /setup-claude -->` markers for idempotent updates
- `apply_setup_claude.py` handles deployment — new templates must integrate with it

## Milestone 4: New Skills

#### Wave 4 (parallel — all 3 skills are independent)

- [x] Create `skills/sk:scope-check/SKILL.md`
  - Read `tasks/todo.md` — extract all planned tasks (checkboxes)
  - Run `git diff main..HEAD --stat` — get list of changed files
  - Compare planned scope vs. actual changes:
    - Planned tasks: count from todo.md checkboxes
    - Actual changes: count files/functions added beyond plan
    - Unplanned additions: changes not traceable to any todo.md checkbox
  - Calculate scope bloat percentage
  - Classify: On Track (<=10%), Minor Creep (10-25%), Significant Creep (25-50%), Out of Control (>50%)
  - Output structured report with recommendations
  - Tools: Read, Glob, Grep, Bash (git commands)

- [x] Create `skills/sk:retro/SKILL.md`
  - Read `tasks/todo.md` — planned tasks
  - Read `tasks/progress.md` — actual progress, errors, resolutions
  - Read `tasks/workflow-status.md` — step-by-step status
  - Analyze `git log` for the feature branch — commit count, files changed, time span
  - Generate retrospective report:
    - Completion rate: planned vs. completed tasks
    - Velocity: commits/day, files changed/day
    - Blockers: extract from progress.md (look for "FAIL", "error", "blocked", "3-Strike")
    - Gate performance: how many gate attempts before clean (from workflow-status.md Notes)
    - Lessons generated: any new entries in lessons.md during this task
    - Action items: 3-5 concrete improvements for next task
  - Output to `tasks/retro-YYYY-MM-DD.md`
  - Check for previous retros — detect recurring patterns
  - Tools: Read, Glob, Grep, Bash (git commands), Write

- [x] Create `skills/sk:reverse-doc/SKILL.md`
  - Accept argument: `<path>` — file, directory, or module to document
  - Phase 1: Analyze — read code, identify patterns, architecture, dependencies
  - Phase 2: Clarify — ask user 3-5 questions to distinguish intent from accident
  - Phase 3: Draft — generate architecture/design doc based on analysis + answers
  - Phase 4: Approve — present draft, get user approval before writing
  - Phase 5: Write — save to `docs/architecture/` or `docs/design/` based on type
  - Flag follow-up work (related areas needing docs) without auto-executing
  - Tools: Read, Glob, Grep, Write, Agent (Explore subagents for deep analysis)

#### Wave 7 (depends on Milestone 6)

- [x] Create `skills/sk:gates/SKILL.md`
  - Single command that runs all quality gates in optimized order:
    - **Batch 1 (parallel agents):** lint + security + perf — spawn 3 agents simultaneously
    - Wait for Batch 1 results. If any agent made fixes, commit is already done (internal to agent).
    - **Batch 2 (agent):** test — needs lint fixes applied first
    - Wait for Batch 2. If fixes needed, agent handles commit internally.
    - **Batch 3 (main context):** review + simplify — needs deep code understanding, stays in main context
    - **Batch 4 (agent):** e2e — needs review fixes applied
  - Report summary after all gates:
    ```
    Gates: lint (clean) | security (clean) | perf (1 fix) | test (clean) | review (2 fixes) | e2e (clean)
    All gates passed. Run /sk:update-task
    ```
  - If any gate fails after 3 attempts → stop and report to user (3-strike protocol)
  - Update `tasks/workflow-status.md` steps 12-17 in one shot
  - Tools: Agent, Read, Write, Bash, Glob, Grep

#### Wave 8 (depends on Milestone 7)

- [x] Create `skills/sk:fast-track/SKILL.md`
  - Abbreviated workflow for small, clear changes:
    1. Read `tasks/todo.md` + `tasks/lessons.md` (quick context)
    2. Branch (`/sk:branch`)
    3. Implement directly (no brainstorm, no design, no TDD)
    4. Commit (`/sk:smart-commit`)
    5. Gates (`/sk:gates` — all quality gates, one command)

    - If diff exceeds 300 lines → warn: "This change looks large. Consider the full workflow."

  - Use cases: config changes, dependency bumps, copy/wording, small refactors, adding missing tests
  - Tools: Read, Write, Bash, Glob, Grep, Agent, Skill

## Risks/Unknowns

- **settings.json template rendering**: The current `apply_setup_claude.py` only handles `.md` templates. JSON templates need careful handling to avoid breaking JSON syntax during placeholder substitution.
- **Hook stdin format**: Claude Code hooks receive JSON on stdin — need to verify exact schema for each event type (SessionStart, PreCompact, PreToolUse, SubagentStart, Stop).
- **Path-scoped rules**: Claude Code's `.claude/rules/` directory may require specific frontmatter format — need to verify the expected structure.
- **Statusline script**: The statusline JSON input format needs to match Claude Code's expected schema (model name, context usage fields).
- **Cross-platform**: Hooks must work on macOS, Linux, and Windows (Git Bash). Use POSIX-compatible patterns, `grep -E` not `grep -P`.
- **Agent model routing**: `.claude/agents/*.md` may need specific frontmatter format for model selection — need to verify Claude Code's agent definition schema.
- **Parallel agent coordination**: `/sk:gates` spawning multiple agents needs careful handling — if agent A fixes a file that agent B also touches, merge conflicts can occur. Batch structure (lint before test) mitigates this.
- **Fast-track scope**: Need clear guidance on when fast-track is appropriate vs. full workflow — user judgment call, but guard rails help.
